## Что сделано на практике
- Создан непривилегированный пользователь с правами `sudo` (отдельного второго администратора решили не заводить)
- Настроен вход по SSH-ключу (ed25519) вместо пароля — ключ сгенерирован на клиенте, публичная часть добавлена в `~/.ssh/authorized_keys` на сервере
- Обнаружено, что хостинг-провайдер по умолчанию уже включил `ufw` с базовыми правилами (80, 443, 22 + порт от предустановленной панели управления, которую не используем). SSH (22) и 80/443 оставлены открытыми (планируется хостинг сайтов), порт неиспользуемой панели оставлен как есть — можно закрыть в любой момент командой `ufw delete <номер>`
- Установлен и настроен `fail2ban` для защиты SSH от подбора пароля: 5 неудачных попыток за 10 минут → блокировка IP на 1 час. Конфигурация вынесена в отдельный файл `/etc/fail2ban/jail.local`, не трогая основной конфиг пакета — чтобы настройки не слетели при обновлении
- Отключены `PasswordAuthentication` и `PermitRootLogin`, проверено через `sudo sshd -T` (эффективная конфигурация) и практическим тестом: вход обычным пользователем по ключу проходит без пароля, `ssh root@ip` получает `Permission denied`

## Результат
Базовая защита внедрена: sudo-пользователь, вход по SSH-ключу, firewall, fail2ban, отключён вход по паролю и под root. Остался один пункт — смена порта SSH с 22 на нестандартный.

## Проблемы и решения
- В Windows PowerShell нет команды `ssh-copy-id` (это Linux/macOS утилита) — ключ добавляли вручную через `echo "публичный_ключ" >> ~/.ssh/authorized_keys` с сервера
- Во время `apt upgrade` пересобрался и перезапустился SSH-сервис, сессия оборвалась — переподключились без потери данных
- Правки `PasswordAuthentication no` / `PermitRootLogin no` в `/etc/ssh/sshd_config` не срабатывали — хостинг-провайдер уже положил `/etc/ssh/sshd_config.d/40-hosting.conf` с явными `yes` для обоих параметров. Файлы из `sshd_config.d/` подключаются в начале главного конфига, и в SSH побеждает первое встреченное значение — пришлось редактировать именно `40-hosting.conf`, а не основной файл. Обнаружили через `sudo sshd -T` — команда показывает реально применяемую конфигурацию, а не то, что написано в конкретном файле
