# Этап 2. Базовая защита сервера

**Статус:** 🟡 в процессе

**Задача:** закрыть очевидные дыры до того, как сервер увидят боты-сканеры (а они найдут его в течение часов после включения).

## Пошагово
- [x] Обновить систему: `apt update && apt upgrade -y`
- [x] Создать обычного пользователя с правами sudo (не работать постоянно под root):
  `adduser ваше_имя` → `usermod -aG sudo ваше_имя`
- [x] Настроить вход по SSH-ключу вместо пароля (для одного администратора):
  - Сгенерировать ключ на своём компьютере (`ssh-keygen`)
  - Скопировать публичный ключ на сервер (`ssh-copy-id`)
- [ ] Настроить SSH-ключ для второго администратора
- [ ] Отключить вход по паролю и вход под root в файле `/etc/ssh/sshd_config`:
  - `PasswordAuthentication no`
  - `PermitRootLogin no`
- [ ] Сменить порт SSH с 22 на нестандартный (снижает шум от автоматических атак)
- [x] Установить `fail2ban` — блокирует IP после нескольких неудачных попыток входа
- [x] Настроить firewall (`ufw`):
  - Разрешить только нужные порты (новый SSH-порт, позже — VPN, SIP)
  - Всё остальное — закрыть

## Что нужно узнать
- [x] Что такое SSH-ключи (публичный/приватный) и почему это безопаснее пароля
- [ ] Как не потерять доступ к серверу, если ошибётесь в конфиге SSH (совет: держите сессию открытой, пока не проверите новую в отдельном окне)
- [x] Базовые команды `ufw`: allow, deny, status

## Что сделано на практике
- Создано два непривилегированных пользователя (для себя и второго администратора), каждому выданы права `sudo`
- Настроен вход по SSH-ключу (ed25519) вместо пароля — ключ сгенерирован на клиенте, публичная часть добавлена в `~/.ssh/authorized_keys` на сервере
- Обнаружено, что хостинг-провайдер по умолчанию уже включил `ufw` с базовыми правилами (80, 443, 22 + порт от предустановленной панели управления, которую не используем). SSH (22) и 80/443 оставлены открытыми (планируется хостинг сайтов), порт неиспользуемой панели оставлен как есть — можно закрыть в любой момент командой `ufw delete <номер>`
- Установлен и настроен `fail2ban` для защиты SSH от подбора пароля: 5 неудачных попыток за 10 минут → блокировка IP на 1 час. Конфигурация вынесена в отдельный файл `/etc/fail2ban/jail.local`, не трогая основной конфиг пакета — чтобы настройки не слетели при обновлении

## Результат
Базовая защита частично внедрена: sudo-пользователи, вход по SSH-ключу для одного администратора, firewall и fail2ban работают. Отключение входа по паролю отложено до настройки ключа для второго администратора.

## Проблемы и решения
- В Windows PowerShell нет команды `ssh-copy-id` (это Linux/macOS утилита) — ключ добавляли вручную через `echo "публичный_ключ" >> ~/.ssh/authorized_keys` с сервера
- Во время `apt upgrade` пересобрался и перезапустился SSH-сервис, сессия оборвалась — переподключились без потери данных
