# Этап 6. Бэкапы

**Статус:** ✅ завершено

**Задача:** не потерять настройки при поломке.

## Пошагово
- [x] Настроить резервное копирование конфигов (SSH, WireGuard, fail2ban, ufw, Zabbix-агент) в архив на самом сервере
- [x] Настроить расписание через `cron`
- [x] Проверить, что бэкап реально восстанавливается (это важнее, чем сам факт его создания)
- [x] Скопировать архив за пределы сервера — на домашний ПК

## Что нужно узнать
- [x] Куда бэкапить — выбрали простой вариант: локально на сервере (с ротацией) + периодическая копия на домашний ПК
- [ ] Разница между полным и инкрементальным бэкапом (используется полный — конфигов немного, инкрементальность не нужна)

## Что сделано на практике
- Написан скрипт `~/backup.sh`, который архивирует ключевые конфиги (`/etc/wireguard`, `/etc/ssh/sshd_config*`, `/etc/fail2ban/jail.local`, `/etc/zabbix/zabbix_agent2.conf`, `/etc/ufw`) в датированный `tar.gz` в `~/backups`, и удаляет архивы старше 14 дней (`find ... -mtime +14 -delete`)
- Скрипт сделан исполняемым (`chmod +x`) и запущен вручную для проверки
- Настроен `cron` (`crontab -e`) на ежедневный запуск в 3:00 ночи, вывод пишется в лог-файл
- Проверено восстановление: архив распакован во временную папку и сверен с оригиналами через `diff -r` — расхождений нет
- Архив скачан с сервера на домашний ПК через `scp -P 2244` (порт указан явно, так как SSH перенесён с 22 на 2244 на этапе 2)

## Результат
При поломке или ошибке конфигурацию можно восстановить из ежедневного архива — как с самого сервера, так и с локальной копии на ПК, если сервер станет полностью недоступен.

## Проблемы и решения
- В архиве бэкапа лежат приватные ключи (WireGuard) — сам файл архива нужно хранить так же бережно, как и оригиналы, не пересылать по незащищённым каналам
- Восстановление проверялось без `sudo` — `diff` выдал `Permission denied` на `/etc/wireguard` (доступ только для root). Сам бэкап это не портит (создавался через `sudo tar`), просто для сверки тоже нужны права: `sudo diff -r ...`
- При копировании архива на ПК через `scp` порт нужно указывать явно (`-P 2244`), так как стандартный порт 22 на сервере закрыт
