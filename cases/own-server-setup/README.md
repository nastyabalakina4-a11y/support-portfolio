# Кейс: Настройка собственного сервера (Alt Linux)

> 🚧 Кейс в процессе. Часть 1 — практика на тестовом стенде — завершена. Часть 2 — настройка арендованного сервера — выполняется, план пополняется по ходу работы.

## Контекст

Для расширения компетенций в сторону системного администрирования я прохожу практику по настройке Linux-сервера: сначала — на тестовом стенде из двух виртуальных машин (VirtualBox, «Альт Рабочая станция» + «Альт Сервер»), чтобы отработать все действия безопасно и без риска для реальной инфраструктуры, а затем — на арендованном сервере для реального сервиса.

---

## Проблема

* Нужно было с нуля освоить администрирование Linux (Alt Linux) — от базовых команд до настройки сетевых служб
* Требовалось построить с нуля сетевую топологию из двух узлов и настроить между ними маршрутизацию
* Нужно было научиться защищать сервер (firewall, ограничение доступа, SSH по ключу вместо пароля)
* Необходимо было развернуть базовые сетевые сервисы (DHCP, DNS), которые нужны для дальнейшего реального проекта

---

## Моя роль

Специалист, самостоятельно осваивающий системное администрирование Linux

* Настройка и администрирование виртуальных машин с Alt Linux
* Построение и настройка сетевой топологии
* Настройка сетевой безопасности сервера
* Развёртывание сетевых служб (DHCP, DNS, SSH)

## Действия

### 1. Базовые навыки работы в системе
* Работа с документацией (`man`, `info`), поиск по ключевым словам
* Поиск файлов (`find`), поиск текста в файлах (`grep`, в т.ч. рекурсивный и с инверсией)
* Архивирование и распаковка данных (`tar`)
* Работа с файлами и каталогами, скрытые файлы/каталоги, права доступа (`chmod`)

### 2. Управление пользователями и доступом
* Создание групп и пользователей с заданными UID/GID и домашними каталогами (`groupadd`, `useradd`)
* Настройка паролей и проверка учётных записей (`passwd`, `id`, `/etc/passwd`)
* Настройка прав доступа к файлам и каталогам (владелец/группа/остальные)
* Включение двухфакторной аутентификации через Google Authenticator (`libpam-google-authenticator`, `pam_google_authenticator.so`)

### 3. Управление программным обеспечением
* Обновление системы и установка/удаление пакетов (`apt-get`, `rpm`, `epm`)
* Поиск и получение информации о пакетах (`apt-cache search/show`)
* Установка стороннего ПО из RPM (Google Chrome) и через `epm play` (AnyDesk)

### 4. Процессы и службы (systemd)
* Просмотр процессов и построение цепочки родительских PID (`ps -ef`)
* Переключение между таргетами (`rescue.target`, `multi-user.target`, `graphical.target`)
* Управление автозапуском служб: `enable`/`disable`/`mask`/`unmask`, проверка статуса (`systemctl status`)
* Отложенное выключение системы (`at`)

### 5. Сеть, маршрутизация и firewall
* Настройка сетевой топологии из двух ВМ через «внутреннюю сеть» VirtualBox
* Настройка IP-адресации на рабочей станции и сервере (два сетевых интерфейса)
* Включение маршрутизации на сервере (`net.ipv4.ip_forward=1`)
* Настройка firewall через `iptables`: разрешение ESTABLISHED/RELATED, блокировка INVALID, доверенные интерфейсы, разрешённые сервисы (SSH, ICMP)
* Настройка NAT (`MASQUERADE`) для доступа клиентов сети в интернет через сервер
* Политики по умолчанию (`DROP` на входящий и транзитный трафик) и сохранение правил (`iptables-save`, автозапуск службы `iptables`)

### 6. Удалённый доступ по SSH
* Проверка и запуск службы `sshd`
* Подключение к серверу по SSH под непривилегированным пользователем (вход под root запрещён)
* Выполнение удалённых команд без интерактивной сессии
* Генерация пары SSH-ключей (`ssh-keygen`) и настройка входа без пароля (`ssh-copy-id`)
* Настройка алиасов подключения через `~/.ssh/config`

### 7. DHCP-сервер
* Установка и настройка `dhcp-server` на выделенном интерфейсе
* Конфигурация подсети, диапазона адресов, шлюза, времени аренды (`dhcpd.conf`)
* Проверка выданных адресов и логов службы (`journalctl`, файл аренд `dhcpd.leases`)

### 8. DNS-сервер (BIND)
* Настройка кэширующего DNS-сервера (`listen-on`, `forwarders`)
* Проверка резолвинга (`nslookup`)
* Выдача клиентам локального DNS через DHCP
* Создание и проверка собственной зоны (`named-checkzone`, `named-checkconf`), проверка A-записей

## Следующий этап: настройка арендованного сервера

Сервер (VPS, Ubuntu 24.04) арендован под реальную задачу — доступ к российским сервисам из-за границы через VPN (WireGuard), в перспективе звонки клиентам с российским номером через IP-телефонию (Asterisk + SIP-транк) и хостинг статичных сайтов (Nginx).

### Стек

- **Хостинг:** VPS (1 CPU, 2 ГБ RAM, 40 ГБ SSD)
- **ОС:** Ubuntu 24.04 LTS
- **VPN:** WireGuard
- **Безопасность:** SSH-ключи, ufw (firewall), fail2ban
- **Планируется:** Asterisk (IP-телефония), Nginx (хостинг сайтов)

### Архитектура

```
[Клиент, Windows]  --WireGuard туннель-->  [VPS, Москва]  -->  Интернет / рос. сервисы
                                                |
                                       (в планах) Asterisk + SIP-транк --> звонки с рос. номером
                                                |
                                       (в планах) Nginx --> статичные сайты
```

Пошаговый план и статус по каждому этапу: [plan.md](plan.md)

### Заметки по безопасности этого репозитория

В этом кейсе и во всех прикладываемых конфигах:
- Реальный IP сервера заменён на плейсхолдер / указан обезличенно
- Приватные ключи (WireGuard, SSH) не публикуются
- Пароли и данные SIP-аккаунтов не публикуются
