# Этап 3. Автообновления и базовый мониторинг

**Статус:** 🟡 в процессе (автообновления и мониторинг на VPS готовы, Zabbix развёрнут и собирает метрики — остались триггеры для практики)

**Задача:** чтобы сервер не "протух" без вашего участия.

## Пошагово
- [x] Настроить автоматические обновления безопасности: `unattended-upgrades`
- [x] Настроить ротацию логов (`logrotate`) — чтобы логи не забили диск
- [x] Поставить `htop` и `ncdu` для просмотра нагрузки и занятого места на диске

### Мониторинг через Zabbix

**Цель:** учебная практика — разобраться, как устроен Zabbix и как он работает, чтобы указать в резюме базовое знание инструмента. Не production-мониторинг 24/7.

**Архитектура:** Zabbix Server (+ БД + веб-фронтенд) ставится на домашний ПК, на VPS — только лёгкий Zabbix Agent. Связь — passive-проверки: Server сам подключается к Agent на сервере (не наоборот), это не требует проброса портов на домашнем роутере.

- [x] На домашнем ПК: установить Zabbix Server, СУБД и веб-фронтенд (через Docker Compose)
- [x] Импортировать начальную схему базы данных, настроить подключение к БД (сделано автоматически образами Zabbix при первом запуске)
- [x] Зайти в веб-интерфейс, сменить пароль администратора по умолчанию
- [x] На VPS: подключить репозиторий Zabbix, установить `zabbix-agent2`
- [x] На VPS: открыть порт 10050/tcp в `ufw`, ограничив доступ по IP домашнего ПК
- [x] Добавить VPS как узел сети (host) в веб-интерфейсе Zabbix, привязать шаблон `Linux by Zabbix agent`
- [x] Посмотреть собираемые метрики (CPU, память, диск, сеть) в Latest data
- [ ] Для практики настроить один-два триггера/оповещения (например, на заполнение диска) — просто чтобы понять, как это работает

## Что нужно узнать

- [ ] Как посмотреть, сколько места на диске и что его ест (`df -h`, `du -sh`)
- [ ] Разница между `zabbix-agent` и `zabbix-agent2` (выбрали agent2 как более новый и лёгкий, детально не сравнивали)
- [x] Базовые понятия Zabbix: host, item, trigger, template, action — на практике разобрались, что это и как связано

## Что сделано на практике
- Установлен и включён `unattended-upgrades` через `dpkg-reconfigure --priority=low unattended-upgrades`
- Проверено, что `logrotate` уже стоит по умолчанию в Ubuntu и работает по расписанию через systemd-таймер (`logrotate.timer`, ежедневный запуск) — донастраивать не потребовалось
- Установлены `htop` и `ncdu` для просмотра нагрузки и занятого места на диске
- Изучены теоретически (без внедрения): критерии, по которым можно оценить, потянет ли сервер конкретный сайт (тип сайта, количество одновременных посетителей); базовые метрики через `uptime` (load average), `free -h`
- Развёрнут Zabbix Server (версия 7.4) на домашнем ПК через Docker Compose — три контейнера: PostgreSQL, zabbix-server-pgsql, zabbix-web-nginx-pgsql
- На VPS установлен `zabbix-agent2` той же версии (7.4) из официального репозитория Zabbix
- Агент настроен на приём подключений только с IP домашнего ПК (`Server=` в конфиге), порт 10050 открыт в `ufw` с ограничением по IP
- В веб-интерфейсе Zabbix добавлен хост VPS и привязан стандартный шаблон `Linux by Zabbix agent` — пошли реальные метрики: память, CPU, диск, сетевые интерфейсы (в том числе автоматически обнаруженный интерфейс WireGuard-туннеля из этапа 4), контрольная сумма `/etc/passwd` для отслеживания несанкционированных изменений пользователей

## Результат
Сервер обновляется автоматически (обновления безопасности), логи ротируются из коробки, есть инструменты для быстрой диагностики нагрузки и диска (`htop`, `ncdu`). Zabbix развёрнут и собирает метрики с VPS в реальном времени — учебная цель (разобраться в инструменте) достигнута. Осталось на практику — настроить пару триггеров/оповещений.

## Проблемы и решения
- После добавления хоста в Zabbix индикатор доступности агента долго оставался серым ("Unknown") без ошибки — оказалось, что без привязанного шаблона (Template) у хоста нет ни одного элемента данных (item), поэтому Zabbix Server ещё ни разу не пытался ничего опросить. Решение — привязать шаблон `Linux by Zabbix agent`, после чего опрос начался
- Firewall на VPS был настроен на приём подключений только с публичного IP домашнего ПК, но проверка агента всё равно не проходила. Диагностика через `Test-NetConnection` показала, что фактическое соединение идёт через интерфейс VPN-туннеля (того самого WireGuard из этапа 4) с внутренним IP туннеля, а не с обычным публичным адресом — потому что на ПК включён full-tunnel VPN до того же самого сервера, и трафик к нему заворачивается в туннель. Решение — добавить в `ufw` дополнительное правило, разрешающее порт 10050 также с внутреннего IP-адреса VPN-туннеля
