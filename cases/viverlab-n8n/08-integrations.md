# 8. Интеграции с ИИ (OpenAI/Anthropic) — обход блокировки РФ

**Статус:** 🟡 в процессе

**Задача:** подключить ChatGPT (OpenAI API) и/или Claude (Anthropic API) для аналитики и работы с контентом.

## Обнаруженная проблема
OpenAI и Anthropic официально блокируют доступ к своим API с IP-адресов России (с 2023 года) — подтверждено практически: прямой запрос с основного VPS (Москва) к `api.openai.com` возвращал `403 Forbidden` даже с корректным API-ключом и оплаченным балансом.

## Решение
Развёрнут собственный reverse-proxy на втором VPS за рубежом:
- [x] Проверено, что второй сервер (уже используется под VPN друга через Docker) может достучаться до `api.openai.com` (`curl` вернул `200`)
- [x] Установлен Nginx на этом сервере, настроен `location / { proxy_pass https://api.openai.com; }` с проксированием заголовка `Authorization`
- [x] Firewall (`ufw`) на прокси-сервере настроен так, чтобы не задеть уже работающий VPN друга (явно разрешены его порты 443 и 36941), порт 80 (наш прокси) открыт только для IP основного сервера
- [x] В n8n, в credential OpenAI, поле **Base URL** изменено с `https://api.openai.com/v1` на `http://<IP_прокси-сервера>/v1` — подключение протестировано успешно (**Connection tested successfully**)
- [ ] Заблокировано на: баланс API-аккаунта OpenAI = $0, нужно пополнить, чтобы протестировать реальный workflow с ChatGPT
- [ ] По той же схеме позже можно подключить и Anthropic (Claude) — те же ограничения по России, тот же прокси подойдёт

## Что нужно узнать
- [x] Anthropic (Claude) блокирует Россию так же, как OpenAI — переключение на Claude само по себе проблему не решает
- [ ] Стоит ли использовать готовый платный прокси-сервис (например, proxyapi.ru и аналоги) как альтернативу собственному серверу — проще в поддержке, но зависимость от стороннего сервиса

## Что сделано на практике
Подробности реализации — см. раздел «Решение» выше. Основные команды:
```
# на прокси-сервере
apt install nginx -y
# конфиг /etc/nginx/sites-available/openai-proxy:
#   location / { proxy_pass https://api.openai.com; proxy_set_header Host api.openai.com;
#                proxy_ssl_server_name on; proxy_set_header Authorization $http_authorization; }
rm /etc/nginx/sites-enabled/default   # убрать конфликт server_name "_"
apt install ufw -y
ufw allow OpenSSH
ufw allow 443            # порт VPN друга — не трогаем
ufw allow 36941          # порт VPN друга — не трогаем
ufw allow from <IP_основного_сервера> to any port 80
ufw --force enable       # --force пропускает y/n подтверждение (полезно при проблемах с раскладкой клавиатуры)
```

## Результат
Инфраструктурная часть интеграции ChatGPT готова и проверена (соединение проходит через прокси). Осталось пополнить баланс OpenAI и провести полноценный тест workflow.

## Проблемы и решения
- `cat > file` с `sudo` не сработал при создании конфига (та же история, что и в [этапе 3](03-https.md)) — использован `cat > file << 'EOF'` без sudo, поскольку в тот момент работали под root напрямую на прокси-сервере (там пользователь по умолчанию root, а не через sudo)
- На прокси-сервере не было `nano` — конфиг создавался напрямую через `cat > file << 'EOF' ... EOF`, без интерактивного редактора
- Подтверждение `ufw enable` (`y/n`) не срабатывало при вводе `y` — судя по всему, из-за раскладки клавиатуры (похожая кириллическая буква вместо латинской). Решение — `ufw --force enable`, пропускает вопрос полностью
- API-ключ OpenAI показывается только один раз при создании — при потере нужно просто создать новый (старый можно отозвать)
