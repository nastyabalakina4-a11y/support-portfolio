# 7. Безопасное обновление

**Статус:** ✅ завершено

**Задача:** возможность безопасно обновлять n8n без потери данных.

## Пошагово
- [x] Задокументировать пошаговую процедуру обновления: бэкап → `docker compose pull` → `docker compose up -d` → проверка, что всё работает
- [x] Проверить процедуру практически: сделать бэкап, обновить образ, убедиться, что workflow/credentials/логин сохранились
- [x] Определить план отката (rollback)

## Что нужно узнать
- [x] Changelog/breaking changes перед крупным обновлением стоит смотреть в официальном репозитории n8n на GitHub (releases) перед обновлением на новую мажорную версию
- [x] Тег `latest` в docker-compose.yml удобен для обучения (всегда самая свежая версия), но для более контролируемого продакшена лучше зафиксировать конкретную версию — сделано (см. ниже)

## Что сделано на практике
Процедура обновления (задокументирована и проверена практически):
1. Сделать бэкап: `~/n8n/backup-n8n.sh`
2. Скачать новый образ: `cd ~/n8n && sudo docker compose pull`
3. Пересоздать контейнер с новым образом: `sudo docker compose up -d`
4. Проверить, что n8n открывается и вход работает: `http://<IP>:5678`

Проверено практически: бэкап создан, `docker compose pull` и `docker compose up -d` выполнены, контейнер пересоздан (`Container n8n-n8n-1 Running`), после этого в браузере n8n открылся без повторного запроса пароля — сессия и данные не потерялись.

Дополнительно версия зафиксирована явно в `docker-compose.yml` (`image: docker.n8n.io/n8nio/n8n:2.30.5` вместо `latest`) — версия узнана командой `docker exec n8n-n8n-1 n8n --version`. Теперь обновление до новой версии — осознанное действие (смена тега в файле), а не то, что может незаметно произойти при любом пересоздании контейнера.

**План отката, если новая версия не заработала:**
1. `cd ~/n8n && sudo docker compose down`
2. Восстановить том из бэкапа (при необходимости): распаковать нужный архив из `~/n8n-backups/` во временную папку, затем скопировать содержимое обратно в volume через тот же приём с вспомогательным контейнером, что и при бэкапе, только в обратную сторону (`docker run --rm -v n8n_n8n_data:/data -v <распакованная_папка>:/backup alpine sh -c "cp -a /backup/. /data/"`)
3. При использовании зафиксированной версии образа (не `latest`) — просто вернуть предыдущий тег в `docker-compose.yml` и снова `docker compose up -d`

## Результат
Обновление n8n проверено на практике: данные, аккаунт и сессия сохраняются при пересоздании контейнера с новым образом. Есть задокументированный план отката на случай проблем.

## Проблемы и решения
_Проблем не возникло — образ уже был актуальным на момент проверки, но сама процедура (pull → up -d) отработала корректно и без потери данных._
