# 6. Резервное копирование

**Статус:** ✅ завершено

**Задача:** регулярный (желательно автоматический) backup базы данных, конфигурации и пользовательских данных n8n.

## Пошагово
- [x] Написать скрипт, который архивирует содержимое Docker-тома `n8n_n8n_data` (через вспомогательный контейнер, без остановки n8n)
- [x] Настроить ротацию бэкапов (не хранить вечно)
- [x] Настроить расписание через `cron`
- [x] Проверить восстановление: распаковать бэкап во временную папку и убедиться, что данные читаемы
- [x] Задокументировать, где именно лежат бэкапы

## Что нужно узнать
- [x] Volume бэкапится через вспомогательный контейнер `alpine`, который монтирует том `n8n_n8n_data` как `/data` (только для чтения по факту использования, сам n8n продолжает работать без остановки) и папку с бэкапами как `/backup`, затем внутри выполняется `tar`
- [x] Копия вне сервера — рекомендовано так же, как и для общего бэкапа сервера (см. [основной кейс](../own-server-setup/06-backups.md)); можно периодически скачивать `~/n8n-backups/` на ПК той же командой `scp`, что использовалась там

## Что сделано на практике
- Скрипт `~/n8n/backup-n8n.sh` архивирует Docker-том `n8n_n8n_data` в `~/n8n-backups/`, удаляет архивы старше 14 дней
- Запуск через `cron` ежедневно в 4:00 (на час позже общего бэкапа сервера, в 3:00 — чтобы не пересекались по нагрузке)
- Восстановление проверено: архив распакован в `/tmp/n8n-restore-test`, все файлы (`database.sqlite`, `config`, логи, `nodes/`) на месте и читаемы

## Результат
Данные n8n (база данных, ключ шифрования credentials, логи) бэкапятся автоматически каждый день, хранятся в `~/n8n-backups/` на сервере, старые копии ротируются. Восстановление подтверждено практически, не только "в теории".

**Расположение бэкапов (для раздела "После установки необходимо предоставить" из ТЗ):** `/home/nastya/n8n-backups/` на сервере.
